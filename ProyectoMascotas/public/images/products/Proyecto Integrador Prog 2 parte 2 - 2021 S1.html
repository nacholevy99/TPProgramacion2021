Condiciones generales

 La fecha  de la entrega final del proyecto integrador será el  Lunes 28 de Junio.  Las presentaciones se harán en forma grupal vía zoom. Queda a criterio de los alumnos si desean agregar recursos o soporte (como PPTs) para sus exposiciones.

El equipo deberá organizarse para que cada integrante de manera equilibrada pueda sin excepción:

  • Contar las funcionalidades incluidas en una o varias secciones de la página.
  • Explicar técnicamente en detalle una sección del código hecha por quien expone. La exposición implica contar que va haciendo el código línea por línea incluyendo los conceptos teóricos que lo respaldan. Serán inválidas las exposiciones que carezcan de sustento teórico y/o no puedan detallar paso a paso lo que sucede. Durante la presentación los profesores podrán hacer preguntas individuales a los alumnos para precisar conceptos o validar información que hayan expresado.

La nota de la presentación oral es individual mientras que la del trabajo en si es grupal. La entrega se realizará mediante repositorio del proyecto en GIthub.

Es importante que utilicen GIT durante el proyecto y no únicamente para la entrega. Estaremos observando el commit log y quienes realizan dichos commits. Esto afecta tanto a la nota del trabajo como la nota de concepto.

Organización de los equipos

Cada equipo deberá administrar la distribución, asignación y cumplimiento de tareas mediante un tablero Trello.

El equipo deberá designar un responsable del seguimiento del tablero para que se mantenga  actualizado. Quien sea designado responsable velará porque cada miembro del equipo cargue y mantenga actualizadas sus tareas. No será él (el responsable designado) quien deba cargar o actualizar las tareas. En caso de incumplimientos deberá reclamar o exigir a cada miembro del equipo que cumpla con cargar las tarjetas y mantenerlas actualizadas.

Consigna General

Para esta etapa el proyecto deberá contar al menos con las funcionalidades de:

  • Administrar productos*.
  • Administrar usuarios*.

(*) Las especificaciones de cada funcionalidad serán detalladas en cada consigna.

El proyecto deberá respetar el patrón MVC aprendido durante la cursada.

La estética del sitio debe mantener una línea en común a través de todas sus páginas. header y footer deben ser comunes en todas las páginas.

El footer deberá ser actualizado con la información correspondiente de cada grupo.

Punto 1 - Base de datos

Deberán entregar junto con el proyecto el archivo SQL (o dump de la base de datos) utilizado en el proyecto y que cumpla con los siguientes requisitos:

  • Creación de la base de datos.
  • Creación de las tablas de la base de datos.
  • Inserción de datos en las tablas correspondientes.
  • Creación de foreign keys.

El archivo SQL debe permitir a los profesores cargar en Workbench (o otro gestor MySQL) la estructura y acceder a los datos de la base que usaron en el proyecto.

Punto 2 - Registración

El catálogo debe permitir la registración de usuarios. El formulario pedirá al menos los siguientes datos y funcionalidades:

  • Nombre de usuario.
  • Email: validar que es un campo obligatorio y que debe ser único (el email ingresado no debe estar repetido en la base de datos). Si el usuario envía el campo vacío o el email ya existe el usuario debe recibir un mensaje especificando el error.
  • Contraseña: validar que es un campo obligatorio y debe tener al menos 3 caracteres. Debe almacenarse en la base de datos de forma encriptada. Si el usuario envía el campo vacío o con menos de 3 caracteres debe recibir un mensaje especificando el error.
  • Fecha de nacimiento.
  • Al crear nuevos usuarios en la base de datos debe plasmarse la fecha de creación en el campo created_at (o createdAt si usan camel case).
  • Al actualizarse registros en la base de datos debe haber un campo updated_at (o updatedAt si usan camel case) que sea modificado cada vez que un registro sea actualizado.
  • Queda a criterio del grupo agregar campos adicionales para los usuarios.

La página de registración será accesible únicamente para los usuarios deslogueados.

Punto 3 - Login

El catálogo debe contar con una página de login que debe cumplir con las siguientes características.

  • El usuario debe poder ingresar utilizando su email y su contraseña.
  • Validar que el email exista en la db. De lo contrario el usuario debe recibir un mensaje especificando el error.
  • Validar que la contraseña sea correcta. De lo contrario el usuario debe recibir un mensaje especificando el error.
  • La página de login debe contar con la posibilidad de recordar el usuario.
  • Una vez que el usuario se haya logueado el header en todas las páginas debe modificarse:
  • Saludando al usuario.
  • Los links de “Registración” y “Login” deben desaparecer del header.
  • Deben verse accesos a “Mi Perfil”, “Logout” y “Agregar producto”.
  • La página de login será accesible únicamente para los usuarios deslogueados.

Punto 4 - Logout

La página debe permitir el logout completo del usuario.

Punto 5 - Buscador de productos

El catálogo debe contar con un buscador de productos. El cual deberá cumplir con las siguientes características:

  • Buscar un producto por su nombre o por su descripción. Tiene que dar resultados para ambas cosas.
  • El resultado de la búsqueda deberá mostrar un listado con los resultados encontrados. En caso de no encontrar resultados debe mostrar un mensaje indicando “No hay resultados para su criterio de búsqueda”.
  • Al clickear en un producto debe mostrarse la página de detalle del producto.
  • En los resultados de búsqueda debe figurar el nombre del usuario que cargó el producto. Clickear sobre el nombre debe llevar al perfil del usuario.

Punto 6 - Perfil del usuario y edición de datos

Dentro de la página de perfil:

  • Los datos de la cantidad total de productos que cargó el usuario y la cantidad total de comentarios que haya realizado deben venir de la base de datos.
  • Se verán todos los productos cargados por el usuario. Los productos deberán tener su correspondiente link para acceder al detalle.
  • Únicamente el usuario logueado dueño del perfil podrá:
  • Ver el ícono de acceso al formulario de edición.
  • Acceder a editar los campos de su perfil. El formulario de edición debe contar con la seguridad suficiente para que un usuario solamente pueda editar su información.

Punto 7 - Home page

La home del sitio deberá mostrar 2 listado de productos ordenados según su fecha de posteo. Cada equipo definirá la cantidad de productos a mostrar en cada sección.

  • Novedades: mostrará los últimos productos agregados de forma descendente. El último producto agregado debe verse en primer lugar.
  • Más comentados: mostrará los productos de acuerdo a la cantidad de comentarios recibidos de manera descendente. Aparecerá primero el producto con más comentarios. (Actualización: esta funcionalidad es OPTATIVA).

Clickear sobre cada producto debe redirigir al detalle del mismo.

En el listado de productos de la Home deberá figurar el nombre de usuario que haya generado cada producto. Lo mismo aplica al resultado de búsqueda de productos.

Cada comentario contará con el nombre del usuario que generó el comentario y el texto del comentario.

El nombre del usuario que generó un comentario debe llevar al detalle de ese usuario al ser clickeado.

Punto 8 - Productos

  1. Solo los usuarios logueados tendrán la opción de crear un producto. El formulario de creación tendrá los campos asignados por los alumnos sin embargo debe contar cómo mínimo con:
  1. Imagen a mostrar.
  2. Texto descriptivo

De forma automática debe almacenarse en la base de datos la fecha de creación del producto (created_at o createdAt).

  1. La página de detalle debe ser accesible por todos los usuarios sin importar si están logueados o no.
  1. La página debe mostrar el nombre del usuario que cargó el producto. Clickear sobre el nombre del usuario debe llevar a la página de perfil del usuario.
  2. Si el producto que se está visualizando en el detalle fue posteado por el mismo usuario que está logueado aparecería el botón de “Borrar Producto”.
  1. Al clickear dicho botón debe borrarse el producto pero previamente se tiene que verificar que el posteo que está siendo borrado pertenece al usuario logueado. En caso de error debe redirigir al usuario a la página de detalle de ese producto.
  1. Si el producto que se está visualizando en el detalle fue cargado por el mismo usuario que está logueado aparecería el botón de “Editar Producto”.
  1. Tras clickear dicho botón aparecerá un formulario para editar los datos del producto. Dicha página solo debe poder ser accesible por el dueño del posteo.
  2. Tras confirmar y enviar el formulario se actualizan los datos del posteo verificando previamente que el posteo pertenezca al usuario logueado.
  1. Comentarios:
  1. Los comentarios se mostrarán de manera descendente: el último comentario recibido debe estar primero en la lista.
  2. Únicamente los usuarios logueados podrán agregar comentarios. El formulario debe contar con la seguridad suficiente para que solo un usuario logueado pueda agregar un comentario. En caso de que un usuario no logueado quiera dejar un comentario la página deberá direccionar al formulario de login.
  3. El comentario debe mostrar el nombre del usuario que comentó. Clickear sobre el nombre del usuario debe llevar al perfil del usuario.
  1. Al crear nuevos productos o comentarios en la base de datos debe plasmarse la fecha de creación en el campo created_at (o createdAt).
  2. Al actualizarse registros de productos en la base de datos debe haber un campo updated_at (o updatedAt) que sea modificado cada vez que un registro sea actualizado.

NOTA: la cátedra se reserva el derecho de modificar o ajustar la consigna si fuera necesario.
